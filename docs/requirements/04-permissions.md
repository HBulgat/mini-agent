# 4. 权限与安全模型

## 4.1 四种权限模式

| 模式 | 触发方式 | 行为 |
|---|---|---|
| 默认 | 无特殊参数 | 读类工具免确认；写文件、shell 执行、网络请求每次询问用户 |
| `--auto-edit` | 启动参数 | 读 + 写文件免确认；shell 执行仍询问 |
| `--yes` | 启动参数 | 全部免确认（**硬黑名单除外**） |
| `--plan` | 启动参数 | 仅允许只读类工具；禁止写文件、删除、shell 执行；只输出方案，不修改用户环境 |

模式可通过 CLI 启动参数指定，运行时可通过 `/mode` 命令切换。

> 注意：「只读类工具」「写类工具」「执行类工具」的具体划分留待设计阶段决定（详见 `08-deferred-to-design.md`）。需求层只锁定模式语义。

## 4.2 白名单 / 黑名单

系统必须支持**三种粒度**的白名单与黑名单：

1. **命令级**：针对 shell 执行工具的命令字符串进行匹配（含黑名单与白名单）。
2. **路径级**：针对文件读 / 写 / 删类工具的目标路径进行匹配。
3. **工具级**：禁用或启用某个具体工具。

规则文件位置**必须可配置**。具体格式与查找规则留待设计阶段。

## 4.3 内置硬黑名单

系统必须**内置**一组硬编码的危险操作黑名单。这些规则**即使在 `--yes` 模式下也必须拒绝执行**，且不可被用户配置文件覆盖。

硬黑名单覆盖的典型场景包括但不限于：
- 递归删除根目录或用户家目录的危险命令
- Fork 炸弹等系统资源耗尽攻击
- 写入或修改系统级关键路径（具体路径清单留待设计阶段）

## 4.4 Web UI 权限交互

- Web UI 中的权限审批**通过对话框内的卡片形式**与用户交互。
- 卡片必须清晰展示：将要执行的工具名、参数、风险等级、批准 / 拒绝两个操作。
- 不使用浮动弹窗（modal popup）形式。

## 4.5 API Key 处理

API Key 仅出现在配置文件中，**禁止出现在任何日志、Trace 输出、CLI 屏幕显示、Web UI 界面之中**。

## 4.6 Skill 脚本的权限管控

Skill 模块（详见 `11-skill.md`）允许 skill 包含可执行脚本（`scripts/` 子目录）。这些脚本的执行**完全复用现有 `bash` 工具**，因此天然受到本章全部权限规则的约束：

- **四种权限模式**对 skill 脚本生效，规则与 `bash` 工具一致：
  - `--plan`：禁止执行
  - 默认：每次执行需用户确认
  - `--auto-edit`：仍需确认
  - `--yes`：免确认（硬黑名单除外）
- **内置硬黑名单**（§4.3）对 skill 脚本同样生效：即使在 `--yes` 模式下，触发硬黑名单的 skill 脚本仍必须被拒绝。
- **白黑名单**（§4.2，命令级 / 路径级 / 工具级）对 skill 脚本生效。

系统**不**为 skill 脚本提供任何绕过权限模型的执行通道。
