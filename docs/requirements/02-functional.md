# 2. 功能性需求

本章按能力域组织。每个能力域只描述**用户可见的行为**，不规定实现方式。

## 2.1 LLM 接入

- 系统**必须支持流式输出**：模型生成的文本应以增量方式呈现给用户，而非整体返回后一次性显示。
- 系统**必须使用大模型的原生 Function Calling 能力**进行工具调用，不采用基于 Prompt 解析的伪工具机制。
- 系统**必须能够接入 DeepSeek-V4-Pro**作为主力模型，接入方式遵循 OpenAI 兼容协议（通过指定 base_url 与 api_key）。
- 系统**必须提供可扩展的 LLM Provider 抽象**，使得未来可以接入其他协议族的模型。规划内置三类 provider：
  - OpenAI 兼容协议（覆盖 DeepSeek、通义、Kimi、SiliconFlow、Ollama 等）
  - Anthropic 原生协议
  - Gemini 协议
- 同一时刻**只激活一个 provider**（单一 active provider 模型）。运行时可通过命令切换激活的模型。
- **请求超时**：LLM 单次请求默认超时 60 秒，超时后视为请求失败。
- **网络层重试**：LLM 请求因网络或限流原因失败时，按等间隔策略最多重试 3 次。
- **业务层重试与网络层重试是相互独立的两层概念**：业务层重试指"工具调用失败让模型重试"，详见 §2.2。
- **Token 与费用统计**：
  - 必须读取 LLM response 中的 usage 字段作为统计来源，**禁止使用字符数粗略估算**。
  - 用户可在以下三处查看统计结果：CLI REPL 内置命令、Web UI 面板、日志文件。
  - 统计粒度必须覆盖：单次请求、单会话累计、全局累计。

## 2.2 Agent 执行循环

### 2.2.1 循环模式

采用**轻规划 + 经典 ReAct** 模式：
- 系统**不内置任务复杂度判断逻辑**。
- 是否制定计划由模型自主决定：模型可以选择直接进入工具调用循环，或先调用 `write_plan` 工具写入计划再推进。
- 工具调用循环遵循 Thought → Tool Call → Observation 的经典 ReAct 形态。

### 2.2.2 步数限制

- 单次任务存在**最大循环步数上限**（用户可见的可配置项）。
- 达到上限时循环必须停止并向用户给出当前进展。

### 2.2.3 工具调用失败的业务层重试

- 单个工具单次调用失败时，agent 应让模型重试。
- 单个工具的失败重试上限为 **3 次**。
- 重试达到上限后，必须向模型注入提示信息，告知"此路径已多次失败，请尝试换一种方式"。
- 工具调用的重试次数与整体循环步数**独立计算**，互不挤占。

### 2.2.4 终止条件

Agent 循环在以下任一条件满足时终止：
1. 模型不再返回新的 tool_call（自然结束）。
2. 达到最大循环步数。
3. 用户中断（参见 §2.6.3 Ctrl+C 行为）。

不引入显式的 `finish` 工具。

## 2.3 子 Agent

- 系统**必须支持子 agent**机制。
- 子 agent 通过专用工具（暂称 `task`）派生，**继承主 agent 的上下文与工具集**。
- 子 agent 调用模式为**同步阻塞**：主 agent 调用子 agent 后等待其完成，再继续后续推理。
- **不允许子 agent 再派生子 agent**：嵌套深度上限为 1。
- **主 agent 与子 agent 的 todo 列表互相可见**。
- 子 agent 失败、超步数或异常返回时，**必须将错误信息暴露给主 agent**，由主 agent 决定下一步动作（重试、换方式或汇报给用户）。
- 不支持角色化预定义子 agent（如 code-explorer、code-reviewer 之类的固定角色），所有子 agent 均为通用同上下文派生型。

## 2.4 工具系统

- 工具体系定位为**完备的最小集合**：覆盖典型编程辅助所需的读、写、搜、执行、规划、派发能力。
- 工具的具体清单、参数定义、与权限模式的对应关系**留待设计阶段确定**（详见 `08-deferred-to-design.md`）。
- 在需求层已确定**必须存在**的工具能力：
  - **规划与 Todo**：模型可调用 `write_plan` 工具自主写入与更新计划，与 todo 列表合一为同一工具。是否再提供 `list_plan` 之类的辅助工具留待设计阶段。
  - **子 agent 派发**：模型可调用 `task` 工具派生同上下文子 agent。

## 2.5 上下文管理

### 2.5.1 自动摘要压缩

- 系统**必须实现自动上下文压缩**，避免长时间会话超出模型上下文窗口。
- 压缩策略具备**可扩展性**：内置至少三类策略——摘要式、滑动窗口式、分层摘要式。
- 用户可在配置层面选择启用何种策略。

### 2.5.2 触发条件

- **自动触发**：当会话累计 token 占用达到上下文窗口的预设阈值时自动启动压缩。
- **手动触发**：用户可通过 CLI 内置斜杠命令 `/compact` 主动触发压缩。

### 2.5.3 保留与截断规则

压缩过程中：
- **必须保留**：系统 Prompt、`AGENTS.md` 内容、最近若干轮对话原文。
- **可截断或摘要**：更早的对话内容、工具调用的参数与结果。

### 2.5.4 跨会话长期记忆

**不在范围内**。系统不在会话之间共享或继承长期记忆。

## 2.6 CLI / REPL

### 2.6.1 启动与输入

- 主形态为**交互式 REPL**：用户启动后进入持续对话循环。
- 支持通过命令行参数传入一次性 prompt（用于一次性任务或管道场景）。

### 2.6.2 内置斜杠命令

REPL 必须支持以下斜杠命令（语义层面要求；具体名称、缩写、参数留待设计阶段微调）：

| 命令 | 行为 |
|---|---|
| `/help` | 显示帮助 |
| `/exit` | 退出 REPL |
| `/new` | 新建会话 |
| `/resume` | 恢复历史会话 |
| `/sessions` | 列出已有会话 |
| `/compact` | 手动触发上下文压缩 |
| `/clear` | 清空当前对话上下文 |
| `/model` | 切换激活模型 |
| `/mode` | 切换权限模式 |
| `/tools` | 查看当前可用工具 |
| `/todos` | 查看当前 todo |
| `/trace` | 切换 Trace 实时显示开关 |
| `/cost` | 查看 token 与费用统计 |
| `/cd` | 切换工作目录 |
| `/skill <name>` | 用户显式触发使用指定 skill（详见 §2.13） |

### 2.6.3 中断与退出

- **第一次 Ctrl+C**：中断当前正在执行的任务，回到 REPL 等待新输入。
- **连续两次 Ctrl+C**：退出 REPL。

### 2.6.4 启动参数

CLI 必须支持以下启动参数（语义层面）：

| 参数 | 行为 |
|---|---|
| 指定配置文件路径 | 用于调试与多套配置切换 |
| 指定模型 | 启动时强制使用指定模型 |
| 指定恢复的会话 | 启动后直接进入指定会话 |
| 指定工作目录 | 覆盖当前 cwd |
| 关闭 trace 实时显示 | 默认行为之外的显式关闭 |
| 一次性 prompt 模式 | 非交互模式，执行完输出结果后退出 |
| 三种权限模式开关 | `--plan` / `--auto-edit` / `--yes`（详见 `04-permissions.md`） |

参数的具体名称与短选项留待设计阶段。

### 2.6.5 输出渲染

- CLI 输出保持**简洁**：流式文本、必要的工具调用提示、错误信息即可。
- **不要求**复杂 Markdown / 高亮 / diff 渲染，避免加重 CLI 实现负担。

## 2.7 Web UI

### 2.7.1 整体形态

- **独立前端项目**，使用 React 框架。
- 前后端分离：mini-agent 后端提供 HTTP API。
- **不做鉴权**：作为本地工具默认开放，无需登录。
- 后端与 CLI **共用同一份 SQLite 数据库**，从而 CLI 与 Web UI 看到同一份会话数据。

### 2.7.2 必备功能（均为 P0）

- 会话列表与切换。
- 模型流式输出的实时渲染。
- Markdown / 代码高亮 / diff 渲染。
- 工具调用过程的可视化展示（含可折叠详情）。
- 文件树侧边栏。
- 文件预览与 diff 查看。
- Todo 面板。
- **权限审批卡片**：通过对话框内卡片形式与用户交互，而非弹窗。
- Trace 面板。
- 配置编辑界面。
- 多 cwd / workspace 切换。

## 2.8 系统 Prompt 与项目级配置

- 系统 Prompt 由产品**固定内置**，不向用户开放完整定制（用户可通过 `AGENTS.md` 注入项目级补充信息）。
- 系统**必须支持项目级配置文件 `AGENTS.md`**：作用类似 Claude Code 的 `CLAUDE.md`，用于注入项目级别的上下文与约定。
- `AGENTS.md` 的查找规则、合并策略、加载时机**留待设计阶段确定**。

## 2.9 会话与持久化

- 系统**必须支持会话持久化**，存储介质为 SQLite。
- **数据库文件路径必须可配置**。
- **REPL 启动行为**：默认新建会话；提供 `/resume` 命令恢复历史会话。
- **不支持会话导出**（如导出为 markdown）。

## 2.10 配置

- 默认配置文件位置为 `~/.mini-agent/config.yaml`。
- 必须支持通过命令行参数指定配置文件路径，便于开发调试与多套配置切换。
- API Key 存储在配置文件中。
- 配置文件的具体字段结构**留待设计阶段确定**。

## 2.11 Trace 与日志

- Trace **以日志为主**，输出到日志文件。
- Trace 必须覆盖以下事件类型：
  - LLM 请求与响应（含 messages、tool_calls、tokens、耗时）
  - 工具调用（输入、输出、耗时、错误）
  - Agent 决策（计划写入、循环步进、子 agent 派发）
  - 上下文压缩事件
  - 权限审批事件
- 用户可通过 CLI 内置 `/trace` 命令切换实时显示开关。
- 日志位置、格式、级别、轮转策略**留待设计阶段确定**。

## 2.12 MCP（Model Context Protocol）

- **仅做 MCP client**，不做 MCP server。
- 支持的传输方式包括：stdio、SSE、Streamable HTTP。
- 优先级**较低**，列入 P3 阶段（详见 `06-phasing.md`），不在 MVP 范围内。

## 2.13 Skill 模块

- 系统**必须支持 Skill 加载机制**：Skill 是模型可按需加载的领域知识包，用于在特定场景下为 agent 注入专业流程、参考资料与可执行脚本。
- Skill 文件规范**完全对齐** Claude Code / CodeBuddy 的 Skill 规范（目录形式，含 YAML frontmatter 的 `SKILL.md`，可选 `scripts/` / `references/` / `assets/`）。
- Skill 通过专用工具 `skill_tool` 由模型主动加载，或通过 `/skill <name>` 由用户显式触发。
- Skill 列入 **P1 阶段**；Web UI Skill 详情面板列入 **P2 阶段**。
- 详细需求见 [11-skill.md](11-skill.md)。
