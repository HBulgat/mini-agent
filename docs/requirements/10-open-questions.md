# 10. 遗留问题与假设

本章记录在多轮需求分析中**未明确闭环**的开放问题，以及分析师在文档落地时所做的合理假设。后续设计或实现阶段如发现假设不符用户意图，需回到需求层修正。

## 10.1 开放问题

- **MVP 范围最终清单**：用户在 N6 中将 MVP 具体范围标记为"设计阶段讨论"。本文档 `06-phasing.md` 给出的 P0 / P1 / P2 / P3 划分是 K 节用户认可的版本，但其中工具具体清单（除 `write_plan`、`task` 外）尚未敲定。设计阶段需要在工具清单确定后回填本文档对应阶段。
- **权限模式下"只读 / 写 / 执行"工具如何分类**：N4 中用户将该问题留待设计阶段。这一分类直接影响 `--plan` 模式可用工具集，需要在设计阶段确定后回填 `04-permissions.md`。
- **`write_plan` 是否再伴随 `list_plan`**：N1 用户认可二合一思路，但 `list_plan` 是否单独存在留待设计阶段。
- **数据库文件默认位置**：D2 用户认可"可配置"，未指定默认位置。设计阶段需选择合理默认。

## 10.2 分析师在文档中所做的假设

为使需求文档完整可用，以下假设在用户未明确表态的情况下被采纳。设计与实现阶段如与用户意图不符需即时澄清：

- **进程重启后不要求恢复正在进行的循环状态**：仅要求历史消息、todo、统计可恢复，不要求 agent 从中断处继续执行。
- **同一时刻单进程仅服务一个主任务循环**：未与用户讨论并发主任务的需求，按单任务串行假设。
- **Web UI 后端可同时服务多个浏览器只读查询**：但写入侧并发处理不在 MVP 范围。
- **`AGENTS.md` 内容必须注入模型上下文**：作为系统 Prompt 的补充。具体注入位置留待设计。
- **日志中不出现 API Key**：作为强制要求纳入安全章节，即便用户未明确说"不要输出 Key"，按通用最小权限假设处理。
- **配置文件之外不存储任何凭据**：未与用户讨论 keychain / 环境变量等替代方案，按 A2/A3/E 假设单一来源。
- **`/trace` 命令是 CLI 实时显示开关**，不影响日志文件落盘行为：日志文件按配置始终落盘。
- **MCP Client 不在 MVP 范围**：用户在 H1 明确"优先级不高"，本文档将其放入 P3。
- **Skill 模块在补丁讨论后纳入需求**：通过 2026-05-23 的需求补丁讨论新增 `11-skill.md`，并联动修订 02 / 04 / 05 / 06 / 07 / 09 / README。Skill 相关的工具参数 schema、目录扫描时机、SKILL.md 解析细节、子 agent 上下文继承的实现方式、压缩后重新注入的具体策略等均**留待设计阶段确定**。

## 10.3 后续修订机制

- 任何对本需求文档的实质性修改（功能新增、范围调整、`07-out-of-scope.md` 中条目反复）必须重新走需求分析流程，并在用户给出"同意需求分析结论"之后才能更新文档。
- 设计阶段对 `08-deferred-to-design.md` 中议题作出的决策，应在设计文档中记录，并在本需求文档对应章节添加交叉引用，但**不应直接重写需求层条款**。
